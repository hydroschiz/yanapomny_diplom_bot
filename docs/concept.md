# Документ-концепция программного продукта `yanapomnyu_bot`

Версия: 0.1 (черновик)  
Дата: 2025-12-12  
Проект: Telegram-бот для хранения и отправки напоминаний (Rust + `teloxide`)  

## 1. Анализ предметной области

### 1.1. Предметная область
Личные и групповые напоминания в мессенджере Telegram: создание напоминаний «своими словами», хранение в базе данных, автоматическая доставка в нужное время с учетом часового пояса пользователя, управление списком напоминаний, а также подписочная модель доступа (платные функции).

### 1.2. Проблемы и боли пользователей
- Разрозненность инструментов (календарь, заметки, таск‑менеджер) и низкая доступность «на ходу».
- Сложность ввода времени/повторений в строгих форматах.
- Ошибки с часовыми поясами, особенно при поездках/релокации.
- Отсутствие «единого места» хранения напоминаний без установки отдельного приложения.

### 1.3. Особенности предметной области для реализации
- Точность времени критична: напоминание должно приходить вовремя (с учетом TZ/UTC).
- Возможны высокие нагрузки в моменты «круглых» времен (например, 09:00).
- Необходима устойчивость к сбоям Telegram API и повторная отправка (retry).
- Важна обратная совместимость со старой MongoDB‑схемой (наследие `legacy_go/`).

## 2. Цели и задачи разработки

### 2.1. Цель
Создать Telegram‑бота на Rust (`teloxide`), который позволяет пользователям быстро создавать, хранить и получать напоминания, обеспечивая:
- удобный ввод естественным языком (через LLM‑сервис),
- надежную доставку напоминаний (scheduler + retry),
- совместимость со старой MongoDB‑БД (данные подписок и напоминаний),
- монетизацию через подписку (YooKassa).

### 2.2. Задачи
- Реализовать основные пользовательские сценарии: настройка TZ, создание/просмотр/удаление напоминаний.
- Интегрировать разбор текста напоминаний через внешний `llm_api` (HTTP).
- Реализовать планировщик отправки напоминаний с атомарным «захватом» задач в MongoDB.
- Реализовать оплату подписки (YooKassa) и хранение статуса подписки в MongoDB.
- Обеспечить совместимость по коллекциям/полям MongoDB со старым ботом.
- Подготовить эксплуатационную документацию (переменные окружения, запуск, зависимости).

## 3. Назначение и общие характеристики продукта

### 3.1. Назначение
`yanapomnyu_bot` — сервис, который принимает сообщения пользователей в Telegram, превращает их в структурированные напоминания, сохраняет в MongoDB и отправляет в нужное время.

### 3.2. Общая характеристика и состав
Проект состоит из нескольких логических компонентов:
- Telegram‑бот (Rust `teloxide`): обработка команд/текста/callback‑кнопок.
- База данных MongoDB (`tgBot`): пользователи, напоминания, записи подписок, транзакции.
- Redis: кэш pending‑платежей и дедупликация webhook‑уведомлений.
- HTTP‑сервер (Rust `axum`): прием webhooks от YooKassa.
- LLM‑сервис (`llm_api`, внешняя Go‑служба): парсинг естественного языка в структуру времени/повторений.

### 3.3. Технологический стек (фактический)
- Язык/рантайм: Rust 2024, `tokio`.
- Telegram: `teloxide`.
- БД: `mongodb` (совместимость с legacy), кэш: `redis`.
- HTTP: `axum`, клиент: `reqwest`.
- Платежи: `yookassa`.

### 3.4. Совместимость с legacy MongoDB (критическое требование)
Сохранены ключевые сущности и поля, используемые старой версией бота (`legacy_go/tgBot/...`):
- БД: `tgBot`
- Коллекции: `users`, `reminds`, `records`, `transactions`
- `users` (пример полей): `id`, `utc`, `timezone`, `delay`, `autodelay`, `morning`, `afternoon`, `evening`, `state`
- `reminds` (пример полей): `id`, `text`, `delay`, `time`, `status`, `remID`, опционально `messageID`, `snoozeTime`, `retryCount`, `retryAt`
- В `reminds` используется legacy‑счетчик автоинкремента `remID`: документ `{ "number": 1, "num": <int> }`
- `records` (пример полей): `id`, `nextPaymentDate`, `active`, `freestate` (+ в новой версии добавлены служебные флаги `expiryWarned`, `remindersDeleted`)

## 4. Типы пользователей, описание и потребности

### 4.1. Основные типы пользователей
- **Обычный пользователь**: создает напоминания «в один чат», хочет максимально простой ввод и точную доставку.
- **Платный пользователь (подписчик)**: ожидает отсутствия ограничений, надежности, сохранности напоминаний и быстрого саппорта.
- **Администратор/поддержка**: нужно видеть ошибки, быстро воспроизводить проблемы, иметь минимальные инструменты диагностики.
- **Разработчик/DevOps**: нужен воспроизводимый запуск (Docker/ENV), понятная архитектура, мониторинг, миграции/совместимость с БД.

### 4.2. Потребности
- Быстрое создание напоминания «своими словами».
- Корректная работа с часовыми поясами.
- Список и управление напоминаниями.
- Прозрачная подписка и оплата.
- Стабильная доставка (в т.ч. при временных сбоях Telegram).
- Приватность: минимизация хранения лишних данных.

## 5. Оценка рынка и аналогов

### 5.1. Классы аналогов
- **Мобильные приложения‑напоминалки/таск‑менеджеры** (Google Calendar/Reminders, Todoist, Microsoft To Do, Any.do): сильны по функционалу и UX, но требуют отдельного приложения и контекста вне Telegram.
- **Telegram‑боты‑напоминалки**: удобны тем, что находятся «там же, где чат», но часто ограничены строгими форматами ввода, слабой работой с повторениями/таймзонами и отсутствием подписочной модели/поддержки.

### 5.2. Конкурентные преимущества целевого решения
- Естественный язык → структурированное напоминание (через LLM) + подтверждение пользователем.
- Архитектура с scheduler‑циклом, атомарным захватом due‑напоминаний и retry‑логикой.
- Сохранение совместимости с legacy MongoDB (плавная миграция без потери данных).
- Подписка и платежи через YooKassa (монетизация).

## 6. Общие функциональные требования

Ниже — «что должно уметь» приложение (в терминах требований).

- **FR-1 Регистрация/инициализация**: пользователь может начать работу командой `/start`.
- **FR-2 Настройка часового пояса**: пользователь может задать UTC‑смещение или город/таймзону (`/utc`, кнопки).
- **FR-3 Настройки пользователя**: пользователь может настроить кнопки отложить (snooze) и авто‑отложить (`/setup` + inline‑клавиатуры).
- **FR-4 Создание напоминаний**: пользователь отправляет текст → бот запрашивает подтверждение перед отправкой в LLM.
- **FR-5 Разбор через LLM**: бот отправляет текст и контекст (TZ + текущая дата/время) в `llm_api` и получает структуру (тип, time_spec, recurrence).
- **FR-6 Подтверждение результата**: бот показывает распарсенное напоминание (текст/время/тип) и предлагает создать/редактировать/отменить.
- **FR-7 Хранение**: подтвержденное напоминание сохраняется в MongoDB в legacy‑совместимом формате (`reminds`, `remID` автоинкремент).
- **FR-8 Просмотр и удаление**: пользователь может получить список активных напоминаний (`/list`) и удалить выбранное.
- **FR-9 Доставка**: фоновый планировщик периодически находит due‑напоминания и отправляет их в Telegram.
- **FR-10 Retry/устойчивость**: при временных ошибках Telegram применяется retry‑стратегия; при постоянных — напоминание помечается как завершенное/ошибочное.
- **FR-11 Подписка**: создание напоминаний доступно только при активной подписке; пользователь может оформить/проверить оплату (`/pay`).
- **FR-12 Webhooks YooKassa**: система принимает webhook‑уведомления и активирует подписку, предотвращая двойную обработку (дедуп в Redis).

## 7. Нефункциональные требования

- **NFR-1 Надежность**: отсутствие дублирования отправок (атомарный захват напоминаний), восстановление после падения процесса.
- **NFR-2 Производительность**: обработка батчей due‑напоминаний и параллельная отправка с ограничением concurrency.
- **NFR-3 Точность времени**: корректные вычисления времени с учетом TZ/UTC и пользовательских «утро/день/вечер».
- **NFR-4 Совместимость**: сохранение схемы MongoDB/полей для работы со старой базой без миграций (по возможности).
- **NFR-5 Безопасность**: секреты (токены/ключи) не должны храниться в репозитории; доступ к БД/Redis по защищенным URI.
- **NFR-6 Наблюдаемость**: структурированное логирование (уровни через `RUST_LOG`), диагностика ошибок внешних сервисов.
- **NFR-7 Поддерживаемость**: модульная архитектура (`api/`, `bot/`, `scheduler/`), единая точка доступа к БД (`Db`).
- **NFR-8 Развертываемость**: запуск локально и в Docker (MongoDB/Redis/LLM API как зависимости).

### Примечания по текущему состоянию репозитория
- `legacy_go/` используется как референс готовой логики (исторический парсинг времени/платежи/схема MongoDB).
- `llm_api` в `docker-compose.yml` ссылается на внешний путь и не входит в этот репозиторий; для воспроизводимого запуска требуется отдельный репозиторий/подмодуль.
- Текущая реализация ориентирована на приватные чаты (фильтры `private_chat_*`); сценарии групп/каналов требуют отдельной проработки.
